3: 1. Introduction to Cryptography

Definition:

  • Cryptography is the practice and study of techniques for secure communication in the presence of third parties (called adversaries).
  • It involves transforming understandable data (plaintext) into an unintelligible form (ciphertext) and back again.
  • Derived from the Greek words “kryptos” (hidden) and “graphein” (to write).

Importance of Cryptography:

  • Confidentiality: Ensures that only authorized individuals can access sensitive information.
  • Integrity: Guarantees that data has not been altered or tampered with during transmission or storage.
  • Authentication: Verifies the identity of users, devices, or systems.
  • Non-repudiation: Prevents a sender from denying they sent a message.
  • Data Security: Protects data at rest (stored) and in transit (being transmitted).
  • Privacy: Safeguards personal and sensitive information from unauthorized disclosure.
  • Trust: Forms the basis of trust in digital interactions, e-commerce, and online services.

4: 2. Basic Concepts of Cryptography

Plaintext and Ciphertext:

  • Plaintext: The original, readable message or data before encryption.
    • Example: “HELLO WORLD”
  • Ciphertext: The scrambled, unreadable message or data produced after encryption.
    • Example (Caesar Cipher, shift 3): “KHOOR ZRUOG”

Key and Algorithm:

  • Algorithm (Cipher): A set of rules or a mathematical function used for encryption and decryption.
    • Functioning: It defines the steps to transform plaintext to ciphertext and vice-versa.
    • Examples: AES, RSA, DES.
  • Key: A piece of information (a parameter) that works with the algorithm to encrypt and decrypt data. The security of the ciphertext depends heavily on the secrecy and strength of the key.
    • Functioning: It customizes the output of the algorithm. Different keys will produce different ciphertexts for the same plaintext when using the same algorithm.
    • Use Cases: Unlocking encrypted data, signing digital documents.
    • Pros: Essential for security; allows multiple secure sessions using the same algorithm.
    • Cons: Key management can be complex; if a key is compromised, security is lost.

Encryption and Decryption:

  • Encryption: The process of converting plaintext into ciphertext using an algorithm and a key.
    • Functioning: The algorithm manipulates the plaintext based on the key.
  • Decryption: The process of converting ciphertext back into plaintext using an algorithm and the corresponding key.
    • Functioning: The algorithm reverses the encryption process using the key.

Cryptographic Protocols and Standards:

  • Protocols: A defined sequence of steps and rules that two or more parties agree upon to communicate securely. They often combine various cryptographic techniques.
    • Functioning: Specify how keys are exchanged, how data is encrypted/decrypted, and how messages are authenticated.
    • Use Cases: SSL/TLS for secure web browsing, PGP for secure email.
    • Pros: Ensure interoperability and standardized security practices.
    • Cons: Can be complex to implement correctly; vulnerabilities in protocol design can compromise security.
  • Standards: Established guidelines and specifications for cryptographic algorithms, protocols, and practices, often set by organizations like NIST (National Institute of Standards and Technology) or ISO (International Organization for Standardization).
    • Functioning: Provide benchmarks for security and interoperability.
    • Use Cases: FIPS 140-2 for cryptographic modules.
    • Pros: Promote strong, tested cryptographic solutions.
    • Cons: Can take time to develop and update, potentially lagging behind new threats.

5: 3. Types of Cryptography - Overview

  • Symmetric Cryptography: Uses a single, shared key for both encryption and decryption.
  • Asymmetric Cryptography: Uses a pair of keys – a public key and a private key.
  • Hash Functions: Uses a mathematical algorithm to transform data into a fixed-size string of characters (hash value). Not for encryption in the traditional sense (irreversible).
  • Hybrid Cryptography: Combines the strengths of symmetric and asymmetric cryptography.

6: 3. Types of Cryptography - Symmetric Cryptography

(e.g., AES, DES, RC4)

Functioning:

  • The sender and receiver must share the same secret key.
  • The sender encrypts the plaintext using the shared key and the chosen symmetric algorithm.
  • The receiver uses the same shared key and algorithm to decrypt the ciphertext back to plaintext.

Key Exchange and Encryption Methods:

  • Key Exchange Challenge: The primary difficulty is securely sharing the secret key between parties without an adversary intercepting it. This often requires an out-of-band method or the use of asymmetric cryptography (like Diffie-Hellman) for initial key exchange.
  • Encryption Methods:
    • Block Ciphers (e.g., AES, DES): Encrypt fixed-size blocks of plaintext at a time. If the message is larger than a block, it’s divided, and modes of operation (e.g., CBC, ECB) are used.
    • Stream Ciphers (e.g., RC4, ChaCha20): Encrypt data one bit or byte at a time. Often faster than block ciphers and suitable for real-time data streams.

Strengths:

  • Speed: Generally much faster than asymmetric cryptography, making it ideal for encrypting large amounts of data.
  • Simplicity: Algorithms are often less computationally intensive.
  • Strong Encryption (with good algorithms like AES): Can provide very high levels of security if the key is sufficiently long and kept secret.

Limitations:

  • Key Distribution: Securely sharing the secret key is a significant challenge. If the key is compromised, all messages encrypted with it are compromised.
  • Key Management: For a large number of users, managing unique shared keys for every pair of users becomes impractical (n(n-1)/2 keys for n users).
  • No Non-repudiation: Since both parties have the same key, you can’t definitively prove who created a message if the key is shared.
  • Scalability: Less scalable for large, open networks due to key distribution issues.

Use Cases:

  • Encrypting files on a hard drive (Full Disk Encryption).
  • Securing database information.
  • Bulk data encryption in SSL/TLS after the initial handshake.
  • Wireless network security (e.g., WPA2/3 using AES).

7: 3. Types of Cryptography - Asymmetric Cryptography

(e.g., RSA, ECC)

Functioning:

  • Involves a pair of mathematically related keys for each user:
    • Public Key: Can be freely shared with anyone. Used to encrypt data or verify a digital signature.
    • Private Key: Kept secret by the owner. Used to decrypt data encrypted with the corresponding public key or to create a digital signature.
  • Data encrypted with a public key can only be decrypted with the corresponding private key.
  • Data signed with a private key can be verified with the corresponding public key.

Public and Private Key Pairs:

  • Generated together. It is computationally infeasible to derive the private key from the public key.

Key Exchange Protocols (e.g., Diffie-Hellman):

  • While asymmetric cryptography can be used for direct encryption, it’s often used to securely exchange a symmetric key, which is then used for bulk encryption (this is hybrid cryptography).
  • Diffie-Hellman Key Exchange: Allows two parties with no prior knowledge of each other to jointly establish a shared secret key over an insecure channel.
    • Functioning: Each party generates a private and public key pair. They exchange public keys. Each party then uses their own private key and the other party’s public key to independently compute the same shared secret.
    • Use Cases: Establishing secure sessions in SSL/TLS, VPNs.
    • Pros: Enables secure key exchange over insecure channels without pre-shared secrets.
    • Cons: Susceptible to Man-in-the-Middle (MitM) attacks if public keys are not authenticated.

Digital Signatures and Certificate Authorities:

  • Digital Signatures:
    • Functioning: The sender creates a hash of the message and then encrypts the hash with their private key. This encrypted hash is the digital signature.
    • The recipient decrypts the signature using the sender’s public key (verifying authenticity) and compares the hash with a newly computed hash of the received message (verifying integrity).
    • Use Cases: Ensuring message authenticity, integrity, and non-repudiation (sender cannot deny sending).
    • Pros: Provides authentication, integrity, and non-repudiation.
    • Cons: Adds computational overhead.
  • Certificate Authorities (CAs):
    • Trusted third-party organizations that issue and manage digital certificates.
    • Functioning: A CA verifies the identity of an entity (e.g., a website, an individual) and then issues a digital certificate that binds their public key to their identity. This helps prevent MitM attacks by confirming the legitimacy of a public key.
    • Use Cases: Validating website identities in HTTPS, verifying software publishers.
    • Pros: Establishes trust in public keys.
    • Cons: If a CA is compromised, the trust model breaks down; can be costly.

Strengths:

  • Secure Key Exchange: Solves the key distribution problem of symmetric cryptography.
  • Digital Signatures: Provides authentication, integrity, and non-repudiation.
  • Scalability: More scalable for large networks as only public keys need to be widely distributed.

Limitations:

  • Speed: Significantly slower than symmetric cryptography, not suitable for encrypting large amounts of data directly.
  • Key Length: Requires much longer keys than symmetric cryptography for equivalent security.
  • Complexity: Algorithms are more computationally intensive.
  • Private Key Security: The private key must be kept absolutely secret; if compromised, all security is lost.

Use Cases:

  • Secure key exchange in SSL/TLS and other protocols.
  • Digital signatures for software and documents.
  • Email encryption (e.g., PGP/GPG).
  • Secure shell (SSH) for remote login.

8: 3. Types of Cryptography - Hash Functions

(e.g., SHA-256, MD5)

Functioning:

  • A hash function takes an input (message or data of any size) and produces a fixed-size string of characters, called a hash value, hash code, digest, or simply hash.
  • It’s a one-way function: easy to compute the hash from the input, but computationally infeasible to generate the original input from its hash value.
  • Any change in the input data, even minor, will result in a significantly different hash value.

Role in Data Integrity and Verification:

  • Data Integrity: By comparing the hash of received data with the original hash (sent separately or stored securely), one can verify if the data has been altered during transmission or storage. If the hashes match, the data is intact.
  • Password Storage: Instead of storing plaintext passwords, systems store their hash values. When a user logs in, the system hashes the entered password and compares it to the stored hash.
  • Digital Signatures: Used to create a concise representation (digest) of a message before it’s signed with a private key.

Characteristics of a Good Cryptographic Hash Function:

  • Pre-image Resistance (One-way): Given a hash value h, it should be computationally infeasible to find an input message m such that hash(m) = h.
  • Second Pre-image Resistance (Weak Collision Resistance): Given an input m1, it should be computationally infeasible to find another input m2 (where m1 ≠ m2) such that hash(m1) = hash(m2).
  • Collision Resistance (Strong Collision Resistance): It should be computationally infeasible to find any two distinct inputs m1 and m2 such that hash(m1) = hash(m2). (MD5 is broken in this regard).
  • Deterministic: The same input must always produce the same hash output.
  • Avalanche Effect: A small change in the input should produce a significantly different hash output.
  • Fixed Output Size: Produces a hash of a consistent length regardless of input size.
  • Efficiency: Easy and fast to compute the hash value for any given message.

Strengths:

  • Integrity Verification: Excellent for ensuring data hasn’t been tampered with.
  • Efficiency: Fast to compute.
  • Password Protection: Provides a secure way to store password representations.

Limitations:

  • Not Encryption: Cannot be used to encrypt and then decrypt data (one-way).
  • Collisions: While theoretically rare for good hash functions, collisions (two different inputs producing the same hash) can exist. Older algorithms like MD5 are known to have practical collision attacks.
  • No Confidentiality: Does not hide the content of the data.

Use Cases:

  • Verifying file integrity (e.g., software downloads).
  • Storing passwords securely.
  • Creating digital signatures.
  • Blockchain technology (linking blocks).
  • Data fingerprinting and duplicate detection.

9: 3. Types of Cryptography - Hybrid Cryptography

(Combining Symmetric and Asymmetric Encryption)

Functioning:

  • Leverages the strengths of both symmetric and asymmetric cryptography while mitigating their respective weaknesses.
  • Typical Process:
    1. The recipient generates an asymmetric key pair (public and private).
    2. The recipient shares their public key with the sender.
    3. The sender generates a random symmetric key (session key).
    4. The sender encrypts the actual message/data using this symmetric key and a fast symmetric algorithm (e.g., AES).
    5. The sender then encrypts the symmetric session key using the recipient’s public key and an asymmetric algorithm (e.g., RSA).
    6. The sender transmits both the encrypted message (encrypted with symmetric key) and the encrypted symmetric key (encrypted with public key) to the recipient.
    7. The recipient uses their private key to decrypt the encrypted symmetric session key.
    8. The recipient then uses the decrypted symmetric session key to decrypt the actual message.

Strengths:

  • Security of Asymmetric Key Exchange: Benefits from the secure key exchange capabilities of asymmetric cryptography for the session key.
  • Efficiency of Symmetric Encryption: Benefits from the speed of symmetric cryptography for encrypting the bulk of the data.
  • Scalability: Well-suited for secure communication between many parties.

Limitations:

  • Complexity: More complex to implement than using either type of cryptography alone.
  • Performance Overhead: Still has some overhead from the asymmetric encryption part, though less than encrypting all data asymmetrically.
  • Dependency on Both Key Types: Security relies on the strength of both the symmetric algorithm and the asymmetric algorithm, as well as the security of both the private key and the symmetric session key.

Use Cases:

  • SSL/TLS (HTTPS): The most common use case. Asymmetric crypto is used for the initial handshake to authenticate the server and securely exchange a symmetric session key. The subsequent web traffic is then encrypted using the faster symmetric key.
  • PGP/GPG Email Encryption: Uses asymmetric keys to encrypt a symmetric session key, which is then used to encrypt the email body.
  • Secure VPNs: Often use asymmetric crypto for authentication and symmetric key exchange, then symmetric crypto for encrypting the data tunnel.

10: 4. Cryptographic Algorithms - Overview

This section will delve into the specifics of common algorithms, their mathematical foundations, and how they provide CIA (Confidentiality, Integrity, Authenticity).

11: 4. Cryptographic Algorithms - AES (Advanced Encryption Standard)

Type: Symmetric Block Cipher

Detailed Explanation:

  • Adopted by the U.S. government to replace DES.
  • Operates on fixed-size blocks of data (128 bits).
  • Supports key sizes of 128, 192, or 256 bits.
  • Uses a substitution-permutation network, not a Feistel network (like DES).
  • Encryption involves multiple rounds of operations, with the number of rounds depending on the key size (10 rounds for 128-bit, 12 for 192-bit, 14 for 256-bit).
  • Each round consists of several steps:
    • SubBytes: A non-linear substitution step where each byte is replaced with another according to a lookup table (S-box).
    • ShiftRows: A transposition step where the last three rows of the state are cyclically shifted.
    • MixColumns: A mixing operation that operates on the columns of the state, combining the four bytes in each column.
    • AddRoundKey: Each byte of the state is combined with a round key derived from the main encryption key.

Mathematical Basis:

  • Operations are performed in a finite field (Galois Field GF(28)).
  • Involves modular arithmetic, byte substitutions, row shifting, and column mixing based on matrix multiplication in the finite field.
  • The key schedule algorithm generates a series of round keys from the main key.

How it Ensures CIA:

  • Confidentiality: Strong encryption makes it computationally infeasible to decrypt ciphertext without the correct key. The multiple rounds and complex operations ensure thorough diffusion and confusion.
  • Integrity (Indirectly, when used with MACs): While AES itself primarily provides confidentiality, it’s often used in modes of operation (like GCM or CCM) that combine encryption with a Message Authentication Code (MAC) to ensure data integrity and authenticity.
  • Authenticity (Indirectly, with MACs): Similar to integrity, authenticity is provided when AES is used in authenticated encryption modes.

Pros:

  • Extremely secure and widely adopted.
  • Resistant to known cryptanalytic attacks when implemented correctly.
  • Efficient in both hardware and software.
  • Flexible key sizes.

Cons:

  • Complexity of implementation can lead to errors if not done carefully.
  • Like all symmetric ciphers, secure key exchange is a prerequisite.

Use Cases:

  • Securing sensitive data at rest (disk encryption, database encryption).
  • Securing data in transit (SSL/TLS, VPNs).
  • Wireless security (WPA2/3).
  • File and archive encryption.

12: 4. Cryptographic Algorithms - RSA (Rivest-Shamir-Adleman)

Type: Asymmetric Algorithm

Detailed Explanation:

  • One of the first practical public-key cryptosystems and still widely used.
  • Based on the presumed difficulty of factoring large prime numbers.
  • Key Generation:
    • Choose two large distinct prime numbers, p and q.
    • Compute n = pq* (modulus). n is used for both public and private keys.
    • Compute Carmichael’s totient function: λ(n) = lcm( p-1, q-1 ).
    • Choose an integer e (public key exponent) such that 1 < e < λ(n) and gcd(e, λ(n)) = 1.
    • Determine d (private key exponent) as d ≡ e-1 (mod λ(n)). d is the modular multiplicative inverse of e modulo λ(n).
  • Public Key: (n, e)
  • Private Key: (n, d) (or sometimes just d, with p and q also kept secret)
  • Encryption (C = Pe mod n): Plaintext P is encrypted to ciphertext C.
  • Decryption (P = Cd mod n): Ciphertext C is decrypted back to plaintext P.
  • Digital Signature:
    • Sign: S = Md mod n (where M is the message hash)
    • Verify: M = Se mod n (and compare with hash of received message)

Mathematical Basis:

  • Number theory, specifically modular arithmetic and prime factorization.
  • The security relies on the fact that it is extremely difficult to factor the large number n into its prime factors p and q if only n and e are known. Knowing p and q allows easy calculation of d.

How it Ensures CIA:

  • Confidentiality: Data encrypted with the public key can only be decrypted with the corresponding private key, which is kept secret.
  • Integrity (via Digital Signatures): When used for digital signatures, it ensures that the message has not been altered, as any change would invalidate the signature.
  • Authenticity (via Digital Signatures): Only the owner of the private key can create a valid signature, thus authenticating the origin of the message.
  • Non-repudiation (via Digital Signatures): The sender cannot deny sending the message if the signature is valid.

Pros:

  • Well-understood and widely implemented.
  • Provides encryption, digital signatures, and key exchange.
  • Strong security with sufficiently large key sizes (e.g., 2048 bits or more).

Cons:

  • Much slower than symmetric algorithms.
  • Large key sizes are required for good security, leading to more computational overhead.
  • Vulnerable to certain attacks if implemented improperly or if key sizes are too small (e.g., factoring attacks, timing attacks).
  • Can be resource-intensive for constrained devices.

Use Cases:

  • Securely exchanging symmetric keys (e.g., in SSL/TLS handshake).
  • Digital signatures for software, documents, and certificates.
  • Email encryption (PGP/GPG).

13: 4. Cryptographic Algorithms - ECC (Elliptic Curve Cryptography)

Type: Asymmetric Algorithm

Detailed Explanation:

  • An approach to public-key cryptography based on the algebraic structure of elliptic curves over finite fields.
  • Offers equivalent security to RSA but with much smaller key sizes, leading to faster computations and lower power consumption.
  • Key Generation: Involves choosing an elliptic curve, a base point G on the curve, and a private key d (a random integer). The public key Q is then computed as Q = dG* (scalar multiplication of point G by d).
  • The core mathematical problem is the Elliptic Curve Discrete Logarithm Problem (ECDLP): given points G and Q on the curve, it is computationally infeasible to find the integer d such that Q = dG*.

Mathematical Basis:

  • Based on the properties of elliptic curves over finite fields. An elliptic curve is defined by an equation (e.g., y2 = x3 + ax + b).
  • Operations involve point addition and point multiplication (scalar multiplication) on the curve.

How it Ensures CIA:

  • Confidentiality (via ECIES - Elliptic Curve Integrated Encryption Scheme): Can be used for encryption, often in a hybrid manner similar to RSA.
  • Integrity (via ECDSA - Elliptic Curve Digital Signature Algorithm): Provides digital signatures to ensure data integrity.
  • Authenticity (via ECDSA): Authenticates the sender through digital signatures.
  • Non-repudiation (via ECDSA): Offers non-repudiation.
  • Also used in ECDH (Elliptic Curve Diffie-Hellman) for key exchange.

Pros:

  • Smaller Key Sizes for Same Security: An ECC key of 256 bits provides comparable security to a 3072-bit RSA key.
  • Faster Computations: Smaller keys lead to faster encryption, decryption, and signature generation/verification.
  • Lower Power Consumption: Ideal for resource-constrained devices like mobile phones and IoT devices.
  • Less Bandwidth and Storage: Smaller keys and signatures require less space.

Cons:

  • More Complex Mathematically: The underlying math is more complex than RSA, which can make implementation harder and more prone to errors if not done by experts.
  • Patents: Historically, some ECC algorithms were encumbered by patents, though many have expired.
  • Less Widespread Adoption (Historically): While growing rapidly, RSA has a longer history of widespread deployment, though ECC is now very common.
  • Susceptibility to certain quantum attacks (similar to RSA): Though typically requires a larger quantum computer.

Use Cases:

  • Mobile device security.
  • IoT device communication.
  • Cryptocurrencies (e.g., Bitcoin, Ethereum use ECDSA).
  • SSL/TLS (often preferred over RSA for performance).
  • Digital signatures and key agreement.

14: 4. Cryptographic Algorithms - DES (Data Encryption Standard)

Type: Symmetric Block Cipher

Detailed Explanation:

  • An older symmetric-key algorithm, once a federal standard in the US.
  • Uses a 56-bit key and operates on 64-bit blocks of data.
  • Based on a Feistel network structure, where the block is split in half, and operations are performed iteratively over 16 rounds.
  • Each round involves substitution (using S-boxes) and permutation.

Mathematical Basis:

  • Feistel structure, substitutions, and permutations. The S-boxes were carefully designed to resist certain types of cryptanalysis known at the time.

How it Ensures CIA:

  • Confidentiality: Intended to provide confidentiality. However, due to its small key size (56 bits), it is now considered insecure against brute-force attacks.
  • Integrity/Authenticity (Indirectly): Like AES, it would need to be combined with a MAC.

Pros:

  • Historically significant and well-studied.
  • Its design principles influenced later ciphers like AES.

Cons:

  • Insecure Key Size: 56-bit key is too short and can be brute-forced with modern computing power relatively easily.
  • Superseded by AES: No longer recommended for new applications.
  • 3DES (Triple DES): An attempt to prolong DES’s life by applying DES three times with two or three different keys. It is slower and still has some theoretical weaknesses compared to AES. While more secure than DES, it’s also being phased out in favor of AES.

Use Cases (Legacy):

  • Historically used in financial applications (e.g., ATMs).
  • Some older systems might still use it, but migration to AES is strongly advised.
  • 3DES might still be found in some payment systems, but its use is declining.

15: 5. Public Key Infrastructure (PKI) - Role

Role of Public Key Infrastructure:

  • PKI is a set of roles, policies, hardware, software, and procedures needed to create, manage, distribute, use, store, and revoke digital certificates and manage public-key encryption.
  • Core Purpose: To establish and maintain a trustworthy networking environment by providing:
    • Authentication: Verifying the identities of users, devices, or services.
    • Confidentiality: Enabling encryption of data.
    • Integrity: Ensuring data has not been tampered with.
    • Non-repudiation: Providing proof of origin and preventing denial of transactions.
  • Problem Solved: How do you trust that a public key actually belongs to the person or entity it claims to belong to? PKI provides this trust mechanism.

16: 5. Public Key Infrastructure (PKI) - Components

Key Components:

  • Certification Authorities (CAs):
    • Functioning: Trusted entities that issue and sign digital certificates. They verify the identity of the certificate requester before issuing a certificate. The CA’s own public key is widely known and trusted (often pre-installed in browsers and operating systems as a “root certificate”).
    • Use Cases: Issuing SSL/TLS certificates for websites, code signing certificates, email certificates.
    • Pros: Central point of trust.
    • Cons: If a CA is compromised, the entire system’s trust can be undermined. Hierarchical CAs (root CA, intermediate CAs) can add complexity.
  • Registration Authorities (RAs):
    • Functioning: Acts as a verifier for the CA. An RA verifies the identity of entities requesting certificates but does not issue certificates itself. It passes validated requests to the CA. Not all PKIs use separate RAs; the CA might perform this function.
    • Use Cases: Offloading identity verification tasks from the CA.
    • Pros: Can distribute the workload of identity verification.
    • Cons: Adds another layer to the process.
  • Digital Certificates:
    • Functioning: Electronic documents that use a digital signature to bind a public key with an identity (e.g., name of an individual, organization, domain name). It contains information like the public key, the identity of the owner, the issuing CA, the validity period, and the CA’s digital signature.
    • The standard format is X.509.
    • Use Cases: Authenticating web servers (SSL/TLS), individuals (email signing/encryption), software publishers.
    • Pros: Provides a verifiable link between an identity and a public key.
    • Cons: Certificates can expire or be revoked; proper management is crucial.
  • Certificate Revocation List (CRL) / Online Certificate Status Protocol (OCSP):
    • Functioning: Mechanisms to check if a certificate has been revoked by the CA before its scheduled expiry date (e.g., if the private key is compromised).
      • CRL: A list published by the CA containing serial numbers of revoked certificates. Clients download and check this list.
      • OCSP: An online protocol allowing a client to query the CA’s server directly about a specific certificate’s status.
    • Use Cases: Ensuring that only valid, uncompromised certificates are trusted.
    • Pros: Allows for timely invalidation of compromised certificates.
    • Cons: CRLs can become large; OCSP can create privacy concerns and single points of failure.
  • Certificate Repository/Store:
    • Functioning: A database or directory where certificates (and CRLs) are stored and can be retrieved.
    • Use Cases: Browsers and operating systems maintain stores of trusted root CA certificates.
    • Pros: Centralized access to certificates.
    • Cons: Must be securely maintained.

17: 5. Public Key Infrastructure (PKI) - Public and Private Keys in PKI

Recap of Roles:

  • Private Key:
    • Role in Encryption/Decryption: Used to decrypt messages that were encrypted with the corresponding public key.
    • Role in Digital Signatures: Used to create a digital signature (by encrypting a hash of the message).
    • Security: Must be kept absolutely secret by the owner. If compromised, the associated identity and security are compromised.
  • Public Key:
    • Role in Encryption/Decryption: Used to encrypt messages intended for the owner of the corresponding private key.
    • Role in Digital Signatures: Used to verify a digital signature created with the corresponding private key.
    • Distribution: Designed to be shared openly. PKI helps distribute and validate these public keys through certificates.

How PKI Utilizes Them:

  1. An entity (e.g., website owner) generates a public-private key pair.
  2. They submit their public key and proof of identity to a Registration Authority (RA) or directly to a Certification Authority (CA).
  3. The RA/CA verifies the entity’s identity.
  4. If verified, the CA creates a digital certificate that includes the entity’s public key, their identity information, and other details (validity period, etc.).
  5. The CA signs this certificate with its own private key.
  6. This signed certificate is then made available (e.g., by the website).
  7. When a user connects to the website, the website presents its certificate.
  8. The user’s browser (or system) uses the CA’s public key (which it already trusts, often pre-loaded) to verify the CA’s signature on the certificate.
  9. If the signature is valid, the browser trusts that the public key in the certificate genuinely belongs to the website.
  10. The browser can now use the website’s public key (from the certificate) to encrypt information (like a symmetric session key) to send securely to the website, which the website can then decrypt with its private key.

18: 5. Public Key Infrastructure (PKI) - SSL/TLS and HTTPS

SSL (Secure Sockets Layer) / TLS (Transport Layer Security):

  • Functioning: Cryptographic protocols designed to provide secure communication over a computer network. TLS is the successor to SSL. They operate at the transport layer (or just above it).
  • How they work (Simplified Handshake):
    1. Client Hello: Client connects to server, proposes SSL/TLS versions, cipher suites, and a random number.
    2. Server Hello: Server chooses SSL/TLS version, cipher suite, sends its own random number, and its digital certificate (containing its public key and signed by a CA).
    3. Certificate Verification (Client): Client verifies the server’s certificate using the CA’s public key (from its trusted root store). Checks validity, revocation status.
    4. Key Exchange:
      • Client generates a “premaster secret” (a random value).
      • Client encrypts this premaster secret with the server’s public key (from the certificate).
      • Client sends the encrypted premaster secret to the server.
    5. Key Derivation: Both client and server independently use the client random, server random, and the premaster secret to derive the same symmetric session keys (for encryption and MAC).
    6. Client Finished: Client sends a “Finished” message, encrypted with the new session key.
    7. Server Finished: Server decrypts the premaster secret with its private key, derives session keys, and sends its own “Finished” message, encrypted with its session key.
    8. Secure Communication: The handshake is complete. All subsequent application data (e.g., HTTP requests/responses) is encrypted and authenticated using the agreed-upon symmetric cipher and session keys.
  • Use Cases: Securing web traffic (HTTPS), email (SMTP over TLS), VPNs, VoIP.
  • Pros: Provides confidentiality, integrity, and server authentication (and optionally client authentication). Widely adopted standard.
  • Cons: Handshake adds latency. Misconfigurations can lead to vulnerabilities. Relies on trusted CAs.

HTTPS (Hypertext Transfer Protocol Secure):

  • Functioning: It’s HTTP layered on top of SSL/TLS. Essentially, HTTPS = HTTP + SSL/TLS.
  • When you see https:// in a URL and a padlock icon in your browser, it means your connection to that website is encrypted and authenticated using SSL/TLS.
  • Use Cases: Secure online banking, e-commerce, login pages, any website handling sensitive information. Increasingly, all web traffic is moving to HTTPS.
  • Pros: All the benefits of SSL/TLS for web communication. Protects against eavesdropping and man-in-the-middle attacks. Boosts user trust and can positively impact search engine rankings.
  • Cons: Slight performance overhead due to encryption/decryption and the handshake (though modern hardware and protocols minimize this). Requires obtaining and managing SSL/TLS certificates.

19: 6. Cryptographic Protocols - Overview

Recap: Protocols define how cryptographic primitives are used to achieve a security goal.

20: 6. Cryptographic Protocols - SSL/TLS (Recap and Focus)

(This was covered extensively under PKI. You can have a brief recap slide here if needed, or refer back to Slide 18. Focus here could be more on the protocol flow if not detailed enough before.)

How Encryption Works in Web Traffic (Summary):

  1. Handshake Phase (Asymmetric Cryptography):
    • Server authentication (using its certificate and private key).
    • Negotiation of cipher suite (which symmetric algorithm, hash function, etc., to use).
    • Secure exchange of a shared secret (premaster secret) using the server’s public key (e.g., RSA) or a key agreement protocol (e.g., Diffie-Hellman).
  2. Bulk Data Transfer Phase (Symmetric Cryptography):
    • Both client and server derive identical symmetric session keys from the shared secret.
    • All subsequent HTTP data is encrypted and decrypted using these fast symmetric session keys (e.g., AES).
    • Message Authentication Codes (MACs) are used with the symmetric encryption to ensure data integrity and authenticity of each message.

Functioning: Provides an encrypted and authenticated channel between a web browser (client) and a web server. Use Cases: Securing nearly all modern web communication (HTTPS). Pros: End-to-end security for web data, server authentication, data integrity. Cons: Complexity, reliance on PKI, potential for misconfiguration.

21: 6. Cryptographic Protocols - IPSec (Internet Protocol Security)

Functioning:

  • A suite of protocols that secures Internet Protocol (IP) communications by authenticating and/or encrypting each IP packet in a data stream.
  • Operates at the network layer (Layer 3 of the OSI model), so it can protect all application traffic.
  • Provides two main security services:
    • Authentication Header (AH): Provides connectionless integrity, data origin authentication, and protection against replay attacks. It does not provide confidentiality (no encryption).
      • How it works: Calculates a hash-based MAC over most of the IP packet fields.
    • Encapsulating Security Payload (ESP): Provides confidentiality (encryption), and can also provide connectionless integrity, data origin authentication, and anti-replay.
      • How it works: Encrypts the IP payload. Can also add an authentication trailer.
  • Two Modes of Operation:
    • Transport Mode: Encrypts/authenticates only the payload of the IP packet. The original IP header is kept. Typically used for host-to-host communication.
    • Tunnel Mode: Encrypts/authenticates the entire original IP packet (header and payload). This encrypted packet is then encapsulated in a new IP packet with a new header. Typically used for site-to-site VPNs (Virtual Private Networks) where gateways handle IPSec.

Key Management: Uses IKE (Internet Key Exchange) protocol (versions IKEv1 and IKEv2) to negotiate security parameters (algorithms, keys) and authenticate peers. IKE often uses Diffie-Hellman for key exchange and digital signatures/certificates for authentication.

Use Cases:

  • Creating Virtual Private Networks (VPNs) to securely connect remote users or branch offices to a corporate network.
  • Securing communication between servers.
  • Providing security for applications that don’t have their own built-in security mechanisms.

Pros:

  • Strong security features (encryption, authentication, integrity).
  • Transparent to applications (operates at the network layer).
  • Flexible (can protect a wide range of traffic).
  • Widely supported standard.

Cons:

  • Complex to configure and manage.
  • Can have issues with Network Address Translation (NAT) traversal (though NAT-T helps).
  • Performance overhead due to cryptographic operations on every packet.
  • AH does not provide confidentiality.

22: 6. Cryptographic Protocols - SSH (Secure Shell)

Functioning:

  • A cryptographic network protocol for operating network services securely over an unsecured network.
  • Primarily used for secure remote login and command execution, but can also be used for secure file transfers (SFTP, SCP) and port forwarding (tunneling).
  • Provides a secure channel between a client and a server.
  • Connection Setup:
    1. Version negotiation.
    2. Server sends its host key (a public key) to the client. Client verifies this key (often by checking against a local list of known hosts to prevent MitM).
    3. Key exchange (e.g., Diffie-Hellman) to establish shared symmetric session keys for encryption and MACs.
    4. User authentication:
      • Password-based: User provides a password (sent over the encrypted channel).
      • Public Key Authentication (more secure): User has a public-private key pair. Their public key is stored on the server. Client proves possession of the private key by signing a challenge from the server.
  • All subsequent traffic is encrypted and authenticated using the negotiated symmetric ciphers and MACs.

Use Cases:

  • Secure remote administration of servers and network devices.
  • Securely transferring files (SFTP, SCP).
  • Tunneling other network protocols (port forwarding) to secure them.
  • Securely accessing version control systems (like Git).

Pros:

  • Strong encryption and authentication.
  • Protects against eavesdropping, connection hijacking, and MitM attacks (if host keys are managed properly).
  • Versatile (remote login, file transfer, tunneling).
  • Public key authentication is very secure.

Cons:

  • Proper host key management is crucial to prevent MitM attacks (users often blindly accept new host keys).
  • Password authentication can be vulnerable to brute-force or weak passwords if not properly secured (e.g., using tools like fail2ban).
  • Can be complex to set up public key authentication for novice users.

23: 6. Cryptographic Protocols - PGP/GPG (Pretty Good Privacy / GNU Privacy Guard)

Functioning:

  • PGP is a commercial encryption program; GPG (or GnuPG) is a free and open-source implementation of the OpenPGP standard.
  • Provides cryptographic privacy and authentication for data communication.
  • Used primarily for signing, encrypting, and decrypting emails, files, and directories.
  • Utilizes a hybrid approach:
    • Encryption:
      1. A random symmetric session key is generated for each message.
      2. The message is encrypted using this session key and a symmetric cipher (e.g., AES).
      3. The session key is then encrypted using the recipient’s public key (RSA or ElGamal/ECC).
      4. Both the encrypted message and the encrypted session key are sent to the recipient.
      5. Recipient uses their private key to decrypt the session key, then uses the session key to decrypt the message.
    • Digital Signatures:
      1. A hash of the message is created.
      2. The hash is encrypted (signed) with the sender’s private key.
      3. This signature is attached to the message.
      4. Recipient uses the sender’s public key to decrypt the hash (verify signature) and compares it with a hash of the received message.
  • Web of Trust: PGP/GPG uses a decentralized trust model. Users can sign each other’s public keys, indicating they trust that key belongs to that person. This differs from the hierarchical CA model of PKI.

Use Cases:

  • Encrypting and signing email messages.
  • Encrypting files for secure storage or transmission.
  • Verifying the authenticity and integrity of software packages.

Pros:

  • Strong encryption and digital signature capabilities.
  • Available for many platforms and integrated into email clients.
  • “Web of Trust” model allows for trust without relying on centralized CAs (can be a pro or con depending on context).
  • GPG is free and open-source, promoting transparency and auditability.

Cons:

  • Can be complex for non-technical users to understand and use correctly (key management, understanding concepts).
  • The “Web of Trust” can be difficult to navigate and establish broad trust.
  • Requires recipients to also use PGP/GPG and have your public key (and you theirs for encrypted replies).
  • Proper private key management is critical.

24: 6. Cryptographic Protocols - The Diffie-Hellman Key Exchange

Type: Key Agreement Protocol (Not an encryption algorithm itself)

Functioning:

  • Allows two parties, Alice and Bob, to establish a shared secret key over an insecure communication channel without any prior shared secrets. This shared secret can then be used as a symmetric key for encryption.
  • Steps:
    1. Agreement on Public Parameters: Alice and Bob agree on two public numbers: a large prime number p (modulus) and a base g (primitive root modulo p). These can be known to everyone, including adversaries.
    2. Alice’s Private and Public Keys:
      • Alice chooses a secret private integer a.
      • Alice computes her public key A = ga mod p.
      • Alice sends A to Bob.
    3. Bob’s Private and Public Keys:
      • Bob chooses a secret private integer b.
      • Bob computes his public key B = gb mod p.
      • Bob sends B to Alice.
    4. Shared Secret Calculation:
      • Alice computes the shared secret: S = Ba mod p = (gb)a mod p = gba mod p.
      • Bob computes the shared secret: S = Ab mod p = (ga)b mod p = gab mod p.
    5. Both Alice and Bob now possess the same shared secret S, which an eavesdropper (who only sees p, g, A, B) cannot easily compute due to the difficulty of the Discrete Logarithm Problem (finding a given g, A, p).

Mathematical Basis:

  • Relies on the difficulty of the Discrete Logarithm Problem (DLP) in modular arithmetic. Given gx mod p, g, and p, it’s hard to find x.

Use Cases:

  • Widely used in many secure protocols to establish session keys, including:
    • SSL/TLS (often as part of ephemeral modes like DHE or ECDHE for Perfect Forward Secrecy).
    • SSH.
    • IPSec (IKE).
    • Secure Real-time Transport Protocol (SRTP).

Pros:

  • Enables two parties to create a shared secret over an insecure channel.
  • Simple and elegant concept.
  • Forms the basis for Perfect Forward Secrecy when used ephemerally (new keys for each session).

Cons:

  • Vulnerable to Man-in-the-Middle (MitM) Attack: If an attacker (Eve) can intercept the communication, she can establish separate keys with Alice and Bob, making them think they are communicating directly.
    • Mitigation: Needs to be combined with an authentication mechanism (e.g., digital signatures from a CA, pre-shared keys, or password-authenticated key exchange) to verify the identities of Alice and Bob and ensure the authenticity of the exchanged public values (A and B).
  • Does not provide encryption or authentication itself; it’s purely a key agreement protocol.
  • Can be computationally intensive with very large prime numbers.

ECDH (Elliptic Curve Diffie-Hellman): A variant that uses elliptic curve cryptography. Provides the same security with smaller key sizes, making it more efficient.

25: 7. Cryptanalysis - Overview

Definition:

  • The study of methods for obtaining the meaning of encrypted information without access to the secret key normally required to do so.
  • It involves analyzing cryptographic systems to find weaknesses or vulnerabilities that could allow unauthorized access to plaintext or compromise the key.
  • The “art and science of breaking codes.”

Goal: To breach the security goals of a cryptosystem (confidentiality, integrity, authenticity).

Types of Attacks (will be detailed):

  • Brute Force
  • Side-Channel Attacks
  • Chosen-Plaintext Attack (CPA)
  • Known-Plaintext Attack (KPA)
  • Ciphertext-Only Attack (COA)
  • Chosen-Ciphertext Attack (CCA)
  • Man-in-the-Middle (MitM) Attack
  • Replay Attack
  • Differential Cryptanalysis
  • Linear Cryptanalysis
  • Birthday Attack (on hash functions)

26: 7. Cryptanalysis - Common Attacks on Cryptographic Systems

  • Brute Force Attack:
    • Functioning: Trying every possible key until the correct one is found.
    • Applicability: Any key-based cipher.
    • Effectiveness: Depends on key length and computational power. Feasible for short keys (like old DES), infeasible for long keys (like AES-256).
    • Pros (for attacker): Simple to implement.
    • Cons (for attacker): Can be extremely time-consuming for strong ciphers.
  • Side-Channel Attacks:
    • Functioning: Exploiting information leaked from the physical implementation of a cryptosystem, rather than theoretical weaknesses in the algorithm.
    • Examples:
      • Timing Attacks: Analyzing the time taken to perform cryptographic operations (which might vary based on the key or data).
      • Power Monitoring Attacks: Analyzing power consumption during operations.
      • Electromagnetic Attacks: Analyzing EM radiation.
      • Acoustic Attacks: Analyzing sounds produced.
      • Fault Analysis: Inducing faults and observing behavior.
    • Applicability: Physical devices performing crypto operations (smart cards, CPUs).
    • Pros (for attacker): Can bypass theoretical strength of algorithms.
    • Cons (for attacker): Requires physical proximity or specialized equipment; complex.
  • Chosen-Plaintext Attack (CPA):
    • Functioning: The attacker can choose arbitrary plaintexts to be encrypted and obtain the corresponding ciphertexts. The goal is to deduce the key or an algorithm to decrypt other messages.
    • Applicability: Situations where an attacker can trick a system into encrypting data of their choice.
    • Significance: A strong cipher should be secure against CPA.
    • Pros (for attacker): Powerful attack model.
    • Cons (for attacker): Requires ability to get chosen plaintexts encrypted.
  • Known-Plaintext Attack (KPA):
    • Functioning: Attacker has access to some plaintext-ciphertext pairs. Goal is to deduce the key used to encrypt other messages.
    • Applicability: Common if standard headers or predictable content is encrypted.
    • Pros (for attacker): More practical than COA if some plaintexts are known.
    • Cons (for attacker): Less powerful than CPA.
  • Ciphertext-Only Attack (COA):
    • Functioning: Attacker only has access to a collection of ciphertexts. This is the most common real-world scenario but often the hardest to exploit.
    • Methods: Statistical analysis, guessing likely plaintexts.
    • Pros (for attacker): Realistic attack scenario.
    • Cons (for attacker): Most difficult, often relies on algorithm weaknesses or very large amounts of ciphertext.
  • Chosen-Ciphertext Attack (CCA / CCA2):
    • Functioning: Attacker can choose arbitrary ciphertexts and obtain their decryptions (plaintext). The goal is to deduce the key. CCA2 (adaptive) allows the attacker to make decryption queries based on previous results.
    • Significance: Considered a very strong attack model. Systems secure against CCA2 are very robust.
    • Pros (for attacker): Very powerful, can reveal subtle weaknesses.
    • Cons (for attacker): Requires access to a decryption oracle.
  • Man-in-the-Middle (MitM) Attack:
    • Functioning: Attacker secretly relays and possibly alters the communication between two parties who believe they are directly communicating with each other.
    • Applicability: Key exchange protocols (like Diffie-Hellman if unauthenticated), SSL/TLS (if certificate validation fails or is bypassed).
    • Pros (for attacker): Can intercept, read, and modify data.
    • Cons (for attacker): Can be detected if proper authentication is used.
  • Replay Attack:
    • Functioning: Attacker intercepts a valid data transmission and maliciously retransmits it later to impersonate a user or repeat an operation.
    • Applicability: Authentication systems, transactional systems.
    • Mitigation: Use of nonces, timestamps, sequence numbers.
    • Pros (for attacker): Simple to execute if system is vulnerable.
    • Cons (for attacker): Easily preventable with proper protocol design.

27: 7. Cryptanalysis - Vulnerabilities in Algorithms

  • Weaknesses in DES:
    • Short Key Length (56 bits): Susceptible to brute-force attacks. Specialized hardware (e.g., EFF’s Deep Crack in 1998) demonstrated this.
    • Susceptibility to Differential and Linear Cryptanalysis: While DES was designed to resist these to some extent, these analytical techniques are more effective against DES than more modern ciphers like AES. These attacks can find the key faster than brute force, though they often require a large amount of known or chosen plaintexts.
  • Attacks on RSA:
    • Factoring Attacks: If the modulus n can be factored into p and q, the private key d can be easily derived. Security relies on the difficulty of factoring large numbers. As computing power increases, key sizes must also increase.
      • Vulnerability: Using too small key sizes (e.g., 512-bit or 768-bit RSA is broken; 1024-bit is deprecated; 2048-bit is common minimum).
    • Small Public Exponent (e) Attacks: If e is small (e.g., e=3) and the same message is encrypted with the same small e under different RSA moduli (n1, n2, n3), the Chinese Remainder Theorem can be used to recover the plaintext (Hastad’s Broadcast Attack).
      • Mitigation: Use proper padding (e.g., OAEP).
    • Implementation Errors:
      • Poor Random Number Generation: If p and q are not truly random or share common factors across different keys, it can lead to vulnerabilities.
      • Side-Channel Attacks: Timing attacks can reveal information about the private key d during decryption operations.
      • Padding Oracle Attacks (e.g., Bleichenbacher’s attack on PKCS#1 v1.5 padding): Allows an attacker to decrypt RSA ciphertexts by sending modified ciphertexts to a server and observing error messages.
        • Mitigation: Use robust padding schemes like OAEP and ensure constant-time implementations.
    • Common Modulus Attack: If two users use the same modulus n but different e and d pairs, and one private key is known, n can be factored. Or, if a message is encrypted with two different public exponents using the same n, it can sometimes be recovered.
  • Weaknesses in Hash Functions (e.g., MD5, SHA-1):
    • Collision Attacks: For MD5 and SHA-1, practical methods have been found to create two different inputs that produce the same hash value (a collision).
      • Impact: Undermines integrity checks and digital signatures if a malicious file can be created with the same hash as a legitimate one.
      • MD5: Severely broken, should not be used for security purposes.
      • SHA-1: Also considered broken for collision resistance; deprecated for most uses (e.g., signing certificates).
    • Pre-image/Second Pre-image Resistance: While MD5 and SHA-1 are still considered somewhat resistant to pre-image attacks (finding input given hash), their collision vulnerabilities mean they should be avoided.

28: 7. Cryptanalysis - Mitigation Strategies and Best Practices

  • Use Strong, Well-Vetted Algorithms:
    • Prefer modern, standardized algorithms like AES (for symmetric), RSA/ECC (for asymmetric, with appropriate key lengths), and SHA-256/SHA-3 (for hashing).
    • Avoid outdated or known weak algorithms (DES, MD5, SHA-1).
    • Stay updated on cryptographic research and algorithm deprecations.
  • Use Sufficiently Long Keys:
    • Key length directly impacts resistance to brute-force attacks.
    • Current recommendations: AES (128-bit minimum, 256-bit preferred), RSA (2048-bit minimum, 3072-bit or 4096-bit for higher security), ECC (256-bit for comparable security to 3072-bit RSA).
  • Secure Key Management:
    • This is often the weakest link.
    • Protect private keys stringently (hardware security modules - HSMs, secure storage).
    • Implement secure key generation, distribution, storage, rotation, and revocation procedures.
    • Use strong random number generators for key creation.
  • Implement Cryptography Correctly:
    • Many vulnerabilities arise from incorrect implementation rather than algorithm flaws.
    • Use well-tested, reputable cryptographic libraries. Avoid implementing crypto primitives yourself unless you are an expert.
    • Follow best practices for using modes of operation (e.g., use authenticated encryption modes like AES-GCM).
    • Properly handle initialization vectors (IVs) and nonces (must be unique, often random).
    • Use secure padding schemes for asymmetric encryption (e.g., OAEP for RSA).
  • Protect Against Side-Channel Attacks:
    • For hardware implementations: use countermeasures like constant-time operations, power/EM shielding, noise generation.
    • For software: be mindful of data-dependent branching or memory access patterns.
  • Implement Proper Authentication:
    • Crucial to prevent MitM attacks, especially with key exchange protocols.
    • Use digital certificates and PKI, or other robust authentication mechanisms.
  • Use Secure Protocols:
    • Employ protocols like TLS, SSH, IPSec that combine cryptographic primitives in a secure manner. Keep them updated to patched versions.
  • Regular Security Audits and Penetration Testing:
    • Have systems reviewed by security experts to identify vulnerabilities.
  • Defense in Depth:
    • Don’t rely on a single layer of security. Combine cryptographic protections with other security measures (firewalls, intrusion detection systems, access controls).
  • Keep Software Updated:
    • Apply patches promptly to address known vulnerabilities in cryptographic libraries and protocols.
  • Educate Users:
    • About phishing, social engineering, and the importance of password hygiene and recognizing secure connections.
